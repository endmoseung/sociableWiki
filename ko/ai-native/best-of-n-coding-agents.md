# Best-of-N 코딩 에이전트 — 토큰을 써서 variance를 사고, 나머지는 버린다

병렬 코딩 에이전트를 다루는 글은 대부분 사실 일을 **나누는** 이야기입니다. 작업 하나를
여러 에이전트에 쪼개고 다시 합치는 거죠. 이 글은 그것과는 **다른** 병렬 패턴을 다룹니다.
2026년 현장이 별도로 이름 붙이고 도구까지 갖춘 패턴, 바로 **best-of-N**입니다. 같은
프롬프트를 격리된 worktree N개에서 중복으로 돌린 다음, **결과물은 딱 하나만 남기고 나머지
N−1개는 버리는** 방식입니다. 병합도 없고, 분해도 없습니다. N배의 토큰을 일부러 써서
*variance*(같은 프롬프트가 만들어 내는 서로 다른 해답의 폭)를 사고, 그중 대부분은 그냥
버리는 겁니다.

## 세 가지 병렬 패턴은 서로 완전히 다르다

가장 흔한 실수는 이걸 전부 "에이전트를 병렬로 돌린다" 한 덩어리로 뭉뚱그리는 것입니다.
세 패턴은 경제성도, 병합 방식도 정반대입니다.

| 패턴 | 무엇이 병렬인가 | 결과물을 어떻게 처리하나 |
|---|---|---|
| **독립 기능** | 서로 다른 *작업* | 전부 병합 — 각각이 별개 기능이니까 |
| **분할-병합** | 한 작업의 서로 다른 *부분* | 하나의 결과로 전부 병합 |
| **Best-of-N** (이 글) | *같은 작업*을 N번 | **하나만 남기고 나머지는 버림** |

병합 단계가 "합치기"가 아니라 "고르기"인 건 best-of-N뿐입니다. Warp 문서는 이걸 정확하게
짚습니다. 같은 작업을 서로 다른 worktree에서 서로 다른 에이전트로 돌린 뒤, **diff를 나란히
놓고 비교해서 승자 하나를 고르라**는 것이죠. 합치라는 게 아닙니다. 같은 문제에 대한 독립된
두 해답을 억지로 병합하면, 두 에이전트 중 누구도 쓰지 않았을 프랑켄슈타인 diff가 나옵니다.

## 왜 되는가: 비결정성은 버그가 아니라 기능이다

분해에서는 LLM의 비결정성(같은 입력에도 매번 다른 출력이 나오는 성질)이 골칫거리입니다. 두
에이전트가 서로 어긋나는 암묵적 선택을 하니까요. best-of-N은 이걸 **뒤집습니다.** 같은
프롬프트가 서로 다른 유효한 해답을 만들어 내고, 우리는 그 폭을 오히려 활용합니다. 여러
자료에서 반복되는 실무상의 이득은 이렇습니다. 한 에이전트는 더 깔끔한 코드를 뽑고, 다른
에이전트는 첫 번째가 놓친 엣지 케이스를 잡아내더라는 것이죠. 작업 전체를 한 번의 뽑기에
거는 대신, 해답 분포에서 여러 번 샘플링해 그중 최댓값을 취하는 셈입니다.

그래서 best-of-N은 **cross-model** 실행과 잘 맞습니다. 같은 프롬프트를 서로 다른 벤더의
에이전트 둘에게 주는 거죠. 한 모델에서 temperature만 바꿔 돌리는 것과는 다릅니다. 모델마다
약점이 다르니 해답의 폭이 더 넓어지고, 그만큼 최댓값도 높아집니다. 벤더 다양성이 좋다는
일반론과 겹치긴 하지만 목적이 다릅니다. cross-check는 "두 에이전트가 한 답의 정합성을
검증"하는 것이고, best-of-N은 "N개 에이전트가 *생성*하고 그중 제일 나은 걸 남기는" 것입니다.

## worktree가 핵심이다 — 같은 격리, 다른 목적

실행 기반은 분해와 똑같습니다. 각 실행마다 자기 git worktree를 하나씩 갖습니다(하나의 공유
`.git` 위에 디렉터리·브랜치·HEAD·dev 서버 포트가 각각 따로). 그래서 N개 실행이 서로 부딪히지
않죠. 요즘 에이전트 도구는 대부분 실행 세션을 각자의 worktree에 자동으로 배치해 주고, 극단적인
형태는 변경 하나를 worktree로 격리된 subagent 여러 개로 부채꼴처럼 펼쳐 각각 PR을 올리는
것입니다. 다만 *의도*가 다릅니다. 분해에서 worktree는 충돌을 진짜 병합 시점으로 미뤄 두는
장치인데, best-of-N에는 **미뤄 둘 병합 자체가 없습니다.** N−1개 worktree는 병합되지 않은 채
삭제됩니다. 뒷정리는 여전히 풀리지 않은 숙제이고(worktree는 스스로 사라지지 않으니까), 그게
그저 패자들 몫으로 넘어갈 뿐입니다.

## 경제성: N배를 치르고, 그 비용은 진짜다

best-of-N은 실제로 출하하는 코드 한 단위당 가장 비싼 병렬 패턴입니다. 버리는 게 설계에
내장돼 있으니까요.

- **토큰 비용은 N에 선형으로 늘고**, 그중 N−1만큼의 토큰 지출은 남는 게 아무것도 없습니다.
  이건 *의도된* 맞바꿈입니다. variance는 공짜가 아닙니다.
- **병목은 사람으로 옮겨 갑니다.** ~2~4개를 넘겨 fan-out하는 순간, 제약은 더 이상 모델이
  아니라 diff를 훑어 승자를 고르는 *나 자신*이 됩니다. 비교·선택 단계는 생성처럼 병렬화되지
  않습니다.
- **선택에는 진짜 기준이 필요합니다.** "더 나아 보이는 걸 고른다"는 후보가 몇 개만 넘어가도
  안 통합니다. N이 클 때는 절대 점수를 매기게 하지 말고 *짝지어 비교*로 순위를 내세요. 두
  diff 중 어느 쪽이 나은지 심사자에게 묻고, 작은 토너먼트를 돌리는 식입니다. LLM 심사자는
  절대 점수 보정이 부정확하니까요. 그리고 승자는 병합 전에 validator 에이전트나 통합
  테스트로 한 번 걸러 주세요. 격리된 상태에서 초록불인 diff가 곧 올바른 diff는 아닙니다.

## Best-of-N이 값어치를 하는 순간(그리고 아닌 순간)

- **값어치를 할 때:** 처리량보다 *최선의* 해답이 중요한, 스펙 잡기 어려운 고위험 작업.
  까다로운 리팩터, 애매한 버그 수정, 설계가 민감한 API 같은 것들이죠. 사람이 안 붙는
  밤샘 실행에도 좋습니다. headless 실행 N개를 걸어 두고, 아침에 그 폭을 훑어보면 됩니다.
- **값어치를 못 할 때:** 아무 정답이나 괜찮은, 스펙이 명확한 일상 변경. 여기서 best-of-N은
  한 번만 돌려도 나올 결과를 두고 토큰만 N배 태웁니다. 저는 기본값을 single-agent로 두고
  (프린스턴 연구에서 단일 에이전트가 멀티 에이전트 구성과 약 64%의 작업에서 대등했습니다),
  어중간한 해답의 대가가 N배 토큰 비용을 넘어설 때만 best-of-N으로 올립니다.

## 한 줄 규칙

분해는 일을 나눠서 전부 병합하고, best-of-N은 일을 **복제해서** 하나만 남깁니다. 같은
프롬프트의 두 결과물을 *병합*하려 애쓰고 있다면 패턴을 잘못 고른 겁니다. best-of-N의 병합
단계는 패자들에게 `git worktree remove`를 거는 것입니다.
