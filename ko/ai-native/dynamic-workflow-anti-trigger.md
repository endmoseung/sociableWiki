# 동적 워크플로우는 기본값이 아니다 — 안 쓸 때를 아는 게 별개의 실력이다

동적 워크플로우(dynamic workflow)를 다루는 글은 대부분 작동 원리를 설명합니다. 에이전트가 짧은 JavaScript coordinator 스크립트를 하나 짜고, 그게 여러 subagent로 fan-out 한다는 것, 그리고 딱 하나 지켜야 하는 불변식(coordinator는 I/O를 하지 않고 부수효과는 전부 agent가 낸다)이 전부죠. 둘 다 "워크플로우가 *어떻게* 돌아가는가"에 대한 답입니다. 정작 매일 쓰다 보면 발목을 잡는 질문에는 답하지 않습니다. **이 작업을 굳이 워크플로우로 만들어야 하나?**

이 빈칸이 중요한 이유는, 도구가 자꾸 "그래, 웬만하면 만들어"라는 쪽으로 등을 떠밀기 때문입니다. 자동 생성 설정을 켜두면 에이전트가 알아서 워크플로우를 짤지 말지 정합니다. 그래서 남발 위험은 사용자의 실수가 아니라 구조적으로 생깁니다. 최근 몇 주 현장 리포트를 보면 백그라운드·병렬 agent가 "데모에서 매일 쓰는 도구로" 넘어왔다고들 하고요. 자동 발동에다 일상 습관까지 겹치면, 2026년 중반의 현실적인 실패 유형은 **더 싼 단계로 충분한데 워크플로우에 손을 뻗는 것**입니다. 필요한데 안 뻗는 게 아니라요.

## 남발이 비싼 이유 — 비용의 모양

"워크플로우가 좀 더 비싸다" 수준의 얘기가 아닙니다. 공식 가이드도 대놓고 말합니다. 동적 워크플로우는 평범한 세션보다 토큰을 *훨씬* 많이 먹고, **한 번 세게 돌리면 한 달치 평소 사용량을 태울 수 있다**고요. 이유는 밑에 깔린 비용 모델에 있습니다. 우리가 돈을 내는 건 spawn 호출이 아니라 **agent 점유(occupancy)** — 몇 개의 agent가 얼마나 오래 살아 있느냐(살아 있는 agent 수 × 시간)입니다. 11일짜리 마이그레이션에 agent 200개를 뿌리는 워크플로우는 *정당한* 극단이고, agent 하나가 두 턴이면 끝낼 일에 8개를 뿌리는 워크플로우는 똑같이 비싼 기계를 돌리면서 나온 순수한 낭비입니다.

그래서 손익분기점은 "이 작업이 큰가?"가 아닙니다. **조율 비용을 상쇄할 만큼 독립적이고 병렬화 가능한 일이 충분히 있고, *동시에* 하나의 context가 스스로 못 채우는 검증 필요가 있는가?** 둘 다여야 합니다. 둘 중 하나가 아니라요.

## 결정 사다리 — 작업을 덮는 가장 낮은 단계를 골라라

비용이 낮은 것부터 높은 것 순으로 네 단계가 있습니다. 실력은 가장 센 걸 집는 게 아니라, 되는 것 중 *가장 낮은* 단계에 손을 뻗는 데 있습니다.

| 단계 | 이럴 때 | 비용의 모양 |
|---|---|---|
| **단일 agent (지금 이 context)** | 일이 하나의 일관된 context에 들어가고, 단계가 순차적이며 서로 의존할 때 | 가장 쌈. context 하나의 토큰 |
| **subagent 몇 개 spawn (flat fan-out)** | 진짜로 독립적인 하위 작업 2~8개를 내가 직접 합칠 때. 결과를 놓고 반복하거나 분기하지 않을 때 | spawn한 context마다 비용. join은 *이* context에서 내가 쥐고 있음 |
| **동적 워크플로우** | 제어 흐름 자체가 결정적이어야 할 때 — 반복, 결과에 따른 분기, fan-out 뒤 다시 fan-out, 결과 집합 across dedup, 마를 때까지 반복 — *그리고* 중간 상태를 내 context에 들고 있으면 흐트러지거나 넘칠 만큼 긴 실행일 때 | 점유 × 시간. checkpoint·재개 가능 |
| **워크플로우 여러 개를 순차로** | 여러 단계짜리 작업(파악 → 설계 → 구현 → 리뷰)이고, 단계 사이에 내가 개입해 있고 싶을 때 | 위의 합. 의도적으로 단계를 나눔 |

"내가 너무 높이 올라갔다"는 신호는 이겁니다. join을 한 문장으로 말할 수 있으면("이 셋 돌리고 내가 합칠게") 워크플로우는 필요 없었던 겁니다. flat fan-out에 직접 합치는 쪽이 더 싸고, 통제권도 내가 쥡니다. 워크플로우가 제값을 하는 건 *스크립트가* 병렬 결과가 시키는 대로 판단을 내려야 할 때뿐입니다. join 자체가 프로그램일 때(거르고, 라우팅하고, 다시 반복), 한 줄짜리가 아닐 때요.

## 구체적인 안티-트리거 — 이럴 땐 워크플로우를 만들지 마라

- **단계가 순차일 때.** B가 A의 출력이 있어야 돌아가는 A→B→C는 단일 agent가 할 일입니다. 병렬로 얻는 게 없고, coordinator는 순수 오버헤드입니다.
- **fan-out이 작고 join이 시시할 때.** 그냥 이어 붙일 독립 조회 3개 → 워크플로우 말고 flat subagent 호출.
- **실행 중간에 보고 조종해야 할 때.** 워크플로우는 백그라운드에서 돌고 끝나야 보고합니다. 턴마다 사람이 방향을 잡아야 하는 작업이라면, 워크플로우는 원하던 조종 채널을 *없애* 버립니다.
- **아직 범위를 안 잡았을 때.** 발견 단계("어느 파일? 어느 화면? diff가 얼마나 큰가?")는 인라인으로 싸게 하는 일입니다. 그걸 *먼저* 하고, 나온 작업 목록이 파이프라인 돌릴 만큼 큰지 그다음에 정하세요. 작업 크기를 *알아내려고* 워크플로우를 짜는 건 순서를 뒤집는 겁니다. 맞는 하이브리드는 인라인으로 정찰한 뒤, 그 결과 위에서 조율하는 겁니다.
- **독립적인 검증 필요가 없을 때.** agent 하나의 출력만으로 정확성이 자명하면, 워크플로우의 핵심 품질 배당인 adversarial 리뷰 기계는 죽은 무게입니다.

## 이게 설정이 아니라 실력인 이유

이 판단은 자동 트리거에 맡길 수 없습니다. 자동 생성은 철저함을 향해 최적화하니, 성향상 사다리를 *올라가는* 쪽으로 기웁니다. 딱 남발이 흐르는 방향이죠. 아래로 누르는 힘은 사람이, 혹은 명시적 예산 아래 조율하는 에이전트가 대야 합니다. "이 작업을 덮는 가장 싼 단계가, 내가 지금 손 뻗으려는 곳보다 정말로 낮은가?" 이건 자동 트리거 일반의 성질이 특수하게 드러난 경우이기도 합니다. 트리거 키워드는 공유된 이름 공간에서 경쟁하고, 가장 능력 있는 해석 쪽으로 남발됩니다. 다만 여기선 그 남발이 소음이 아니라 토큰으로 비쌉니다.

제가 기본으로 삼는 실전 규칙은 이겁니다. **워크플로우 바로 아래 단계에서 시작하고, 일이 나를 밀어 올릴 때만 올라간다 — 절대 꼭대기에서 시작하지 않는다.** 워크플로우는 광범위한 버그 사냥, 큰 마이그레이션, 여러 표면을 훑는 보안 감사, 주장 검증이 붙은 딥 리서치에는 맞는 도구입니다. "이 몇 가지 해주고 알려줘"에는 틀린 도구고요. 워크플로우를 짜기 전에 "flat fan-out에 내가 직접 합치는 걸로 되지 않나?"라는 질문을 모든 워크플로우가 통과해야 하는 관문으로 삼으세요.
