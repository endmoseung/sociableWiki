# 강제 장치 쌓기는 바닥이지 천장이 아니다 — 장기 drift는 전용 탐지기가 필요하다

에이전트를 단단하게 만드는 표준 레시피에는 은근한 약속이 하나 깔려 있습니다. hook을 걸고
(PreToolUse gate, PostToolUse check), memory 레이어를 고정하고, 필요한 규율을 발동시키는
skill을 심어두면 에이전트가 *레일 위에 머무를 것*이라는 약속이죠. 절반만 맞는 말입니다. 이
레시피는 한 턴의 **바닥**은 올려줍니다. 하지만 에이전트가 며칠짜리 작업에서 얼마나 멀리
벗어날 수 있는지, 그 **천장**은 막지 못합니다. 이 둘은 서로 다른 보장인데, 같은 것으로
착각하는 순간 함정에 빠집니다.

## 현장 증거

2026-05-19에 올라온 한 실무자의 자기 보고
([anthropics/claude-code#60506](https://github.com/anthropics/claude-code/issues/60506))는
제목부터 거의 고백에 가깝습니다. *"hook + memory + skill을 전부 걸어뒀는데도 고객 프로젝트에서
6일간 아키텍처가 drift했다"*는 내용이죠. 설정이 부실했던 게 아닙니다. 우리가 권장할 법한 풀
스택 그대로였습니다. 그런데도 6일에 걸쳐 천천히 흘러갔고, 그 어떤 개별 hook이나 skill 호출도
잡아내지 못했습니다. 바로 이 지점이 핵심입니다. 매 턴은 자기 로컬 체크를 통과했는데도, *누적된
궤적*은 원래 의도했던 아키텍처에서 벗어나 있었던 겁니다.

## 왜 매 턴 gate로는 drift를 못 잡나

PreToolUse gate, skill trigger, acceptance-criteria check — 이것들은 전부 **로컬한, 매 턴짜리**
장치입니다. "*지금 이* 편집/도구 호출이 허용되는가 / 올바른가?"에 답할 뿐이죠. drift는 어느 한
턴의 속성이 아닙니다. *시퀀스*의 속성입니다. 편집 하나하나는 저마다 방어 가능하지만, 40번째
턴이 3번째 턴이 세워둔 경계를 슬그머니 다시 정의했고, 80번째 턴은 그 40번째 턴 버전 위에
쌓입니다. 어느 단계에서도 대놓고 불법인 일이 벌어지지 않았으니, 로컬 체크는 발동하지 않습니다.

이건 단위가 안 맞는 문제(granularity mismatch)이고, 독립적인 per-run validator가 매 턴 rubric을
이기는 이유와 같습니다. rubric은 모든 단계에서 통과해도 실행 전체는 실패할 수 있으니까요. drift는
그 단위 불일치가 *정확성* 축이 아니라 *아키텍처* 축에서 벌어진 것입니다.

## 빠진 장치: drift 탐지기를 별도 루프로

해법은 "hook을 더 붙이자"가 아닙니다. 로컬 gate를 더한다고 전역 보장이 되지는 않으니까요. 빠진
조각은 **별도의 주기적 conformance check**입니다. 코드베이스의 현재 상태를 *얼어붙은 아키텍처
의도*(승인된 plan, ADR, 경계 규칙)와 대조하고, 턴 루프와는 다른 주기로 돌아가는 검사죠.

- **다른 입력** — 지금 대기 중인 도구 호출 하나가 아니라, 누적된 diff / 현재 모듈 그래프를
  읽습니다.
- **다른 주기** — 매 도구 호출마다가 아니라, N개 태스크마다 혹은 태스크 이음매(seam)마다
  돌립니다. drift는 몇 턴 정도 쌓이게 두는 건 싸고, 매 턴 검사하는 건 비쌉니다.
- **다른 oracle** — 진실의 출처로 plan/ADR을 두고 대조합니다. 아키텍처 결정을 코드에 파묻지
  말고 plan 위에 적어둬야 하는 이유가 여기 있습니다. 의도가 안정된 장소에 적혀 있지 않으면
  탐지기가 diff할 대상 자체가 없어서, drift는 구조적으로 탐지 불가능해집니다.
- **실패 시 다른 행동** — "이 편집을 막아라"가 아니라 "궤적이 휘었다. 멈추고 다시 앵커를 잡거나
  re-plan하라"입니다. 편집 단위의 거부권이 아니라, 아키텍처 단위의 되감기(rewind)죠.

이진 통과/실패를 넘어 더 나아가고 싶다면, drift를 하나의 이벤트가 아니라 실행 전반에 걸친 측정
가능한 지표로 다룰 수도 있습니다. 다만 실무적으로 챙길 건 더 좁습니다. **분석 단위가 턴이 아니라
실행(run)인 검사를 반드시 돌려야 한다**는 것입니다.

## 이게 정리해주는 경계

하네스 장치를 두 무더기로 나눠봅시다. 매 턴짜리 결정론적 강제 장치 — hook과 gate — 는 튼튼하고
꼭 필요하지만, 그 범위는 *"이 연산이 허용되는가"*에 한정됩니다. 장기 아키텍처 일관성은 *세 번째*
것입니다. 매 턴짜리 정책도 아니고, 모델이 며칠에 걸쳐 알아서 유지해주는 것도 아닙니다. 명시적이고
주기적인 탐지기가 필요합니다. "hook + memory + skill을 다 걸었으니 drift 저항성도 딸려온다"고
여기는 게 바로 범주 오류입니다. 바닥은 높였지만, 천장은 여전히 열려 있습니다.

## 요약

- hook + memory + skill 풀 스택은 **매 턴의 바닥**을 올릴 뿐, **장기 아키텍처 drift**를 막지
  못합니다. 현장 보고: 풀 스택 아래에서 6일간 drift
  ([#60506](https://github.com/anthropics/claude-code/issues/60506)).
- 매 턴 gate는 **로컬한** 장치이고, drift는 **시퀀스**의 속성입니다. 매 턴은 다 통과하는데도
  궤적이 휩니다. 매 턴 rubric이 통과해도 실행 전체는 실패하는 것과 같은 단위 불일치죠.
- 빠진 장치는 **별도의 주기적 conformance check**입니다. 누적 상태를 읽고, 태스크 이음매에서
  돌고, *얼어붙은 plan/ADR*과 diff하며, 실패하면 편집 하나를 막는 대신 다시 앵커를 잡거나
  re-plan합니다.
- 전제 조건: 아키텍처 의도가 안정된 장소(plan/ADR)에 적혀 있어야 합니다. 그렇지 않으면 diff할
  대상이 없어서 drift는 구조적으로 탐지 불가능해집니다.
