# fan-out 스코프 게이트 — 배치를 띄우기 전에 필터를 매니페스트로 얼려라

## 한 줄 요약

fleet 작업에서 가장 위험한 순간은 실행 *도중*이 아니라, **필터가 실제 작업 대상으로 바뀌는 그 한 번의 전환**입니다. 사람이 "열려 있는 lint PR 전부 고쳐"라고 말하면, coordinator 에이전트는 tool 호출 딱 한 번이면 *지금 이 순간* 그 필터가 가리키는 무엇이든 상대로 워커 N개를 띄워버립니다. 이때 값싸면서도 핵심이 되는 통제 장치가 **인터뷰 게이트**입니다. 워커를 하나라도 띄우기 전에 coordinator가 (1) 필터를 실행하고, (2) **풀린 정확한 목록**을 사람에게 되보여주고, (3) 사람에게 그 목록을 확인받고, 그제서야 (4) 그것을 **claim 매니페스트**로 얼립니다. batch ID, 워커별 lane, branch/worktree, 만료 시각까지 담아서 모든 워커가 이 매니페스트를 읽게 합니다. 어떤 워커도 살아있는 필터를 상대로 띄우지 않습니다. 얼려둔 목록을 상대로만 띄웁니다.

## 왜 필터가 위험한 원재료인가

필터(`label:lint`, `is:open author:bot`, `all files in src/`)에는 fan-out에 넘겨주면 안 되는 세 가지 성질이 있습니다.

1. **작성 시점에 범위가 안 잡힌다.** 개수를 알 수 없습니다. "열린 lint PR 전부"는 오늘은 3건이지만 의존성 한번 올리고 나면 내일은 40건입니다. 폭발 반경은 사람이 머릿속에 그린 것이 아니라, 워커가 발사되는 순간 쿼리가 돌려주는 것입니다.
2. **얼어 있지 않고 살아 있다.** 워커 둘이 같은 필터를 다시 평가해 같은 항목을 둘 다 집을 수 있고(이중 작업, merge 충돌), 사람이 "응"이라고 한 뒤 워커가 뜨는 사이에 항목이 슬쩍 들어오거나 빠질 수 있습니다.
3. **공격자가 닿을 수 있다.** 필터의 입력이 issue/PR/comment 본문이라면 — GitHub 위에서 배치를 돌릴 때는 대개 그렇습니다 — 신뢰할 수 없는 텍스트가 워커를 몇 개나, 무엇을 상대로 돌릴지를 결정합니다. 이건 prompt injection 가드레일이 지키는 것과 같은 신뢰 불가 입력 경계인데, 여기서는 injection이 *한* 에이전트의 행동을 바꾸는 게 아니라 에이전트의 *개수*를 바꿉니다. 그래서 자세는 분명해야 합니다. GitHub issue/PR/comment 본문과 PR branch 변경은 신뢰할 수 없는 입력으로 다루고, 이것이 **에이전트 설정이나 sandbox 설정을 절대 덮어쓰지 못하게** 해야 합니다.

게이트가 하는 일은 딱 하나입니다. 사람 앞에서 필터를 목록으로 **단 한 번** 접고, 그 뒤로는 워커가 살아있는 필터를 다시 보지 못하게 하는 것.

## 인터뷰가 실제로 끌어내는 것

인터뷰는 "정말요? (y/n)"이 아닙니다. 짧은 요청 속에 암묵적으로 깔려 있어서 그냥 두면 워커들이 서로 부딪칠 것들을 밖으로 꺼내 못 박습니다.

| 항목 | 무엇을 못 박나 | 건너뛰면 생기는 문제 |
|------|----------------|----------------------|
| **정확한 대상 목록** | 필터 → 열거된 issue/PR ID. 와일드카드는 *탐색*에만 허용, 작업 대상에는 절대 불가 | 범위 없는 fan-out. 사람은 본 적도 없는 개수를 승인해버림 |
| **lane 배정** | 어느 워커가 어느 항목을 맡는지 + **쓰기 범위 배제** | 워커 둘이 같은 파일을 건드림 → merge가 뒤엉킴 |
| **branch / worktree** | 워커당 branch 하나 + worktree 하나 | working tree 공유 → 상태 손상 |
| **claim 메타데이터** | batch ID, session/thread ID, **만료 시각(보통 2~4시간)** | 오래된 claim이 영영 안 풀림. 죽은 워커의 항목이 계속 잠긴 채 남음 |
| **승인된 goal 프롬프트** | 모든 워커가 실행할 정확한 goal 텍스트를, 띄우기 *전에* 검토 | 잘못 적힌 지시를 워커 N개가 자신 있게 실행. 덜 명세된 브리프 하나가 N배로 복제됨 |

만료 시각은 사람들이 잊는 부분입니다. TTL 없는 claim은 곧 데드락 예약입니다. 30% 지점에서 죽은 워커가 자기 항목을 붙들고 있어서, 누군가 손으로 배치를 풀어주기 전까지 안 놓습니다. 매니페스트는 **락(lock)이 아니라 리스(lease)** 입니다. 진척도에 적용하던 checkpoint-by-consequence(결과가 남는 지점마다 체크포인트) 원칙을, 이번엔 *소유권*에 똑같이 적용한 것입니다.

## fleet 오케스트레이션 안에서의 위치

이건 흔한 fleet 노트와는 결이 다른 층, *앞단*의 게이트입니다.

- **실행 중 패턴보다 앞선다.** ReAct / plan-and-execute / delegation 같은 패턴은 *fleet가 이미 돌기 시작한 뒤*의 행동을 다스립니다. 스코프 게이트는 fleet가 **애초에 시작하느냐, 무엇을 상대로 시작하느냐**를 다스립니다.
- **덜 명세된 브리프와는 축이 다르다.** 그쪽 실패는 orchestrator→worker 브리프 하나가 너무 짧은 것, 한 lane 안의 *품질* 문제입니다. 스코프 게이트는 **사람→coordinator** 경계에 있고, 그 실패는 *개수*(워커 수가 틀림)입니다. 둘은 겹칩니다. 모호한 브리프를 틀린 *개수*의 워커에게 보내는 게 최악입니다.
- **"띄워야 하나?" 판단을 더 날카롭게 한다.** 그 질문이 "띄운다 — *이 얼린 목록*을 상대로, *이 lane*으로, *그때* 만료되게"로 바뀝니다.

## 규칙

> 필터는 **탐색**용이지 **작업**용이 아니다. 필터를 풀고 → 사람에게 정확한 목록을 보여주고 → 그것을 매니페스트(batch ID, 워커별 lane, branch/worktree, 만료)로 얼리고 → 매니페스트를 상대로 워커를 띄운다. 어떤 워커도 살아있는 필터를 다시 평가하지 않는다.

같은 원칙에서 나오는 운영상 따름정리 둘.

- **워커는 issue를 만들지 않는다.** coordinator만 만들고, 그 과정에서 **중복을 제거**합니다. 안 그러면 워커 N개가 같은 발견을 N번 올립니다. fan-out될 수 있는 side effect는 전부 하나뿐인 coordinator를 통해 되돌립니다.
- **리뷰 발견은 자동 병합이 아니라 분류 대상이다.** blocker를 넘어가려면 사람이 actionable한 발견을 명시적으로 waive해야 합니다. 앞단의 게이트(스코프)에는 뒷단의 게이트(merge)가 짝으로 붙습니다. 둘 다 자율적으로 도는 가운데 구간을 사람이 확인하는 지점입니다.

## 비용 / 언제 건너뛰나

인터뷰는 배치 앞에 왕복 한 번을 더합니다. 작업 대상이 (a) 필터에서 나왔거나, (b) 대략 3건보다 많거나, (c) 신뢰할 수 없는 콘텐츠에서 나온 순간부터는 그 값어치를 합니다. 손으로 직접 다 적은, 이미 아는 1~2개 대상 목록일 때만 건너뛰세요. 그때는 얼릴 필터가 없으니 게이트가 아무 일도 안 하는 no-op입니다.
