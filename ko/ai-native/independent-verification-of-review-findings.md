# 리뷰 지적은 다른 검토자가 반증하지 못할 때까지 가설이다

AI 코드 리뷰어는 그럴듯한 지적을 빠르게 만든다. 빠짐없이 후보를 찾는 데는 유용하지만
바로 게시하기에는 위험하다. 줄 번호까지 정확히 짚은 코멘트도 caller가 값을 정리하거나,
test가 반대 동작을 증명하거나, feature flag가 실행 경로를 바꾸거나, 다른 파일에서 이미
예외를 처리하고 있으면 틀린 지적이 된다.

따라서 리뷰 지적은 결과가 아니라 가설로 다룬다.

## 발견과 검증을 분리한다

지적을 처음 만든 리뷰어는 자신의 설명에 이미 끌려간다. 같은 리뷰어에게 다시
확인하라고 하면 기존 주장을 더 정교하게 설명하는 데 그치기 쉽다. 다른 리뷰어에게
주장과 관련 코드를 주고, 그 주장을 기각할 수 있는지 확인하게 한다.

검증자는 다음을 확인한다.

- 주변 코드와 관련 caller
- test, type, schema, configuration
- 문제의 failure condition이 실제로 도달 가능한지
- 다른 변경 파일이 이미 해당 경우를 처리하는지
- 주장한 impact가 근거에서 실제로 이어지는지
- 코멘트 위치가 문제를 보여주는 변경 줄인지

판정은 세 가지다.

| 상태 | 뜻 | 게시 기준 |
|---|---|---|
| `CONFIRM` | 도달 가능한 실패와 영향이 근거로 확인됨 | 게시 가능 |
| `REJECT` | 코드나 계약이 주장을 반증함 | 게시하지 않고 이유 보관 |
| `UNVERIFIED` | 현재 근거로 결론을 낼 수 없음 | 결함이 아니라 미확인 범위로 보고 |

## 기각된 지적도 남겨야 하는 이유

거짓 양성을 조용히 버리면 리뷰 파이프라인의 품질을 알 수 없다. 기각 이유를 남기면
컨텍스트 부족, 오래된 가정, framework 동작 오해, 심각도 과장처럼 반복되는 실패를
찾을 수 있다. 이 기록은 다음 prompt와 rubric을 고치는 근거가 된다.

## 게시 가능한 근거의 형태

줄 번호만으로는 부족하다.

```text
주장 → 도달 가능한 실패 조건 → 관찰 가능한 영향 → 이를 뒷받침하는 근거
```

하나라도 빠지면 검증된 결함이 아니라 조언이나 의심에 가깝다.

## 비용을 쓸 경계

별도 검증은 리뷰 한 번만큼의 비용이 추가된다. 자동 GitHub 댓글, 큰 변경, 낯선 코드,
보안 관련 주장처럼 거짓 양성의 비용이 큰 곳에서 특히 가치가 있다. 작은 수동 리뷰라면
사람이 두 번째 검토자 역할을 해도 된다.

Tool receipt가 주장한 작업의 실행 여부를 증명한다면, 독립 검증은 주장한 결함이 반증
시도를 견뎠는지 증명한다.
