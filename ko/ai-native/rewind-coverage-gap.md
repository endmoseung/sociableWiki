# Claude Code `/rewind`은 *에이전트가 무엇을 했는지*가 아니라 *어떤 도구가 바꿨는지*로 되돌린다

`/rewind`(checkpoint 기능, 2.1.x 라인부터 정식 제공, v2.1.191부터 `/clear`까지 인식)은 이름만
보면 만능 undo처럼 보입니다. "코드나 대화를 이전 프롬프트 시점으로 되돌린다"니까요. 그런데
아닙니다. undo가 커버하는 범위는 **변경을 만들어낸 도구**에 묶여 있고, 그 범위는 에이전트가 실제로
바꾼 것들보다 훨씬 좁습니다. 이걸 범용 undo로 믿는 순간 함정에 빠집니다.

## 정확히 어떻게 동작하나

- checkpoint는 **매 편집 직전**에 찍히고, **모든 사용자 프롬프트**가 새 checkpoint를 시작합니다.
  따로 설정할 것 없이 기본으로 켜져 있습니다. 백업은 세션별 file-history 디렉터리에 쌓이고,
  incremental 방식이라 바뀐 파일만 새 버전이 생기며, 세션과 함께 **30일 뒤** 정리됩니다.
- rewind 메뉴는 내가 보낸 프롬프트를 하나씩 나열하는데, 각 지점마다 **성격이 다른 두 종류의
  동작**을 줍니다. 메뉴는 하나로 묶여 있지만 건드리는 상태는 서로 다릅니다.

| 동작 | 디스크를 건드림? | 컨텍스트를 건드림? | 언제 쓰나 |
|---|---|---|---|
| 코드 + 대화 복원 | 예 | 예 | 그 프롬프트 시점으로 완전히 되돌리기 |
| 대화 복원 | 아니오 | 예 | 지금 파일은 두고 대화만 되돌리기 |
| **코드만 복원** | **예** | 아니오 | 대화의 이해는 살리고 잘못된 실행만 다시 하기 |
| 여기서부터 요약 | 아니오 | 예(압축) | 곁가지 스레드를 버리고 앞쪽 맥락은 유지 |
| 여기까지 요약 | 아니오 | 예(압축) | 세팅 과정을 압축하고 최근 작업은 유지 |

이 중에서 가장 쓸모 있으면서 가장 눈에 안 띄는 게 **코드만 복원**입니다. 에이전트의 *머릿속 모델*은
맞는데 *편집*만 틀렸을 때, 파일은 되돌리고 대화는 그대로 두는 거죠. 그러면 에이전트가 이해를 처음부터
다시 세우지 않고, 같은 이해 위에서 구현만 다시 합니다. 제가 커스텀 하네스에서 자주 쓰는
"findings는 남기고 verdict만 지운다"라는 수를 기본 기능으로 넣어둔 셈인데, 여기서는 훅으로 직접
엮을 필요 없이 메뉴 한 줄로 됩니다.

## 발목 잡는 경계: 추적되는 것 vs 아닌 것

checkpoint가 추적하는 건 **Claude의 파일 편집 도구(Write / Edit / NotebookEdit)가 만든 변경뿐**입니다.
에이전트가 그 밖에 한 일은 전부 rewind 눈에 안 보입니다.

| 무엇이 바꿨나 | `/rewind`으로 되돌려지나? |
|---|---|
| Edit / Write / NotebookEdit | **예** |
| `bash` — `rm`, `mv`, `cp`, `>` 리다이렉트, 코드 생성, 마이그레이션, `git` | **아니오** |
| Claude 밖에서 내가 직접 한 편집 | 아니오(같은 파일이면 예외) |
| 동시에 돌던 다른 Claude 세션 | 아니오(같은 파일이면 예외) |
| 외부 효과 — push된 브랜치, 보낸 메시지, DB 쓰기, 배포된 변경 | 아니오(설계상 애초에 범위 밖) |

즉 안전망의 모양은 *어떤 primitive가 그 쓰기를 했는지*가 정하고, 에이전트의 blast radius(피해
반경)는 *그 작업이 무엇을 했는지*가 정합니다. 이 둘은 bash와 외부의 부수 효과가 사는 지점에서 정확히
갈라지는데, 웬만큼 굵직한 작업이라면 되돌릴 수 없는 피해는 대부분 바로 거기에 몰려 있습니다.
`rm -rf build/`, `git reset --hard`, `prisma migrate`, `terraform apply`, `npm publish` — 전부
bash를 거치고, 전부 rewind로 못 되돌립니다. 에이전트의 손을 넓게 열어줄수록(bash allowlist를 넓게,
auto 모드로) 안전망과 피해 반경 사이의 틈은 더 벌어지는데, `/rewind`은 자기가 더 이상 나를 지켜주지
못한다는 신호를 전혀 주지 않습니다.

## 이게 한 층 아래의 같은 원리인 이유

트랜잭션 복구에 관한 오래된 이야기가 있습니다. DB rollback은 스냅샷을 복원하지만, 에이전트 복구는
거기까지 못 가는 경우가 대부분입니다. 에이전트의 단계들이 단일 트랜잭션 밖의 것들을 건드렸기
때문이죠. `/rewind`은 그 이야기의 구체적인 실물입니다. file-history 스냅샷이 바로 그 "DB
rollback"이고, 그건 트랜잭션이 걸리는 한 면(편집 도구)만 커버합니다. bash 부수 효과와 외부 호출은
"트랜잭션 밖" 단계라서, rewind가 아니라 compensation(보상) 계획이 처리해야 합니다. 원리 자체는
새롭지 않습니다. 여기서 못 박히는 건, 실제 출시된 기능이 딱 그 선을 따라 경계를 긋는다는 점이고,
그래서 이걸 놀랄 일이 아니라 체크리스트 항목으로 삼을 수 있다는 것입니다.

이건 짚어둘 만한 durability 규칙의 읽기 쪽 거울상이기도 합니다. 그 규칙은 "*잃었을 때의
결과(consequence-of-loss)*를 기준으로 checkpoint를 찍어라"입니다. `/rewind`은 좁은 면 위에서
*반사적으로*(프롬프트마다, 편집마다) checkpoint를 찍습니다. 정작 결과가 크고 되돌릴 수 없는 쓰기는
바로 그게 못 보는 것들이죠. 다시 하기 싼 면은 반사적으로 checkpoint하면서 되돌리기 비싼 면은 조용히
건너뛰는 것 — 피하고 싶은 딱 그 뒤집힘입니다. 그런데 여기서는 그게 도구의 성질이라, 규율은 운영자가
챙길 수밖에 없습니다.

## 메뉴 뒤에 있는 지렛대가 복원만 있는 게 아니다

같은 `/rewind` 화면에는 **요약(여기서부터 / 여기까지)**도 붙어 있는데, 이건 파일을 *하나도 안
바꿉니다*. 표적을 정한 `/compact`, 즉 undo가 아니라 컨텍스트 창을 다루는 지렛대죠. 메뉴는 하나인데
축은 서로 상관없는 둘입니다. *복원*은 쓰기 쪽 상태(디스크 + 대화)를 옮기고, *요약*은 읽기 쪽
상태(컨텍스트)만 옮깁니다. 이 둘을 뭉뚱그리면("컨텍스트 좀 비우게 rewind해야지") 엉뚱한 도구를 고르는
겁니다. 스레드를 잃지 않으면서 컨텍스트를 비우려면 요약을 쓰고, 원래 세션은 그대로 두면서 다른 접근을
시도해 보려면 문서가 가리키는 건 rewind가 아니라 **fork**(`claude --continue --fork-session`)입니다.
fork는 가지를 치고, rewind는 덮어씁니다. rewind는 그냥 clear/compact 지렛대에는 없는 *과거의 특정
지점을 겨냥한 소급 압축*을 하나 더 얹어줄 뿐입니다.

## 그래서 결론

1. **`/rewind`은 undo가 아니다. "편집 도구가 쓴 것들의 undo"다.** 넓은 범위의 변경을 이걸로
   되돌리려 하기 전에 이 사실을 소리 내어 확인하세요.
2. **마일스톤에서는 commit하고, 그 사이는 rewind로 오간다.** 공식 가이드도 같습니다. checkpoint는
   로컬 undo, git은 영구 기록입니다. 30일 정리에 세션 단위 범위라, rewind는 기록이 아니라 세션 안에서
   쓰는 편의 장치입니다.
3. **작업의 bash·외부 발자국이 클수록 `/rewind`이 지켜주는 몫은 작아진다.** 파괴적이거나 바깥으로
   나가는 실행 전에 기대야 할 안전 장치는 checkpoint 메뉴가 아니라 깨끗한 git 상태(또는 제대로 된
   compensation 계획)입니다.
4. **내가 옮기는 상태가 무엇인지로 지렛대를 고른다.** 파일이면 → 복원, 컨텍스트면 → 요약, 남겨둘지도
   모를 갈래면 → fork. 두 개가 `/rewind` 메뉴를 같이 쓴다는 이유만으로 서로 바꿔 쓸 수 있는 게
   아닙니다.
