# 코딩 에이전트를 자율적으로 만드는 건 권한 승인이 아니라 샌드박스다

## 뒤집어 보기

직관적으로 떠올리는 모델은 이렇습니다. "자율성이 커질수록 위험도 커지니까, 더 빨리 가려면
안전 프롬프트를 꺼야 한다." 그런데 이건 거꾸로입니다. 2026년에 Claude Code와 OpenAI Codex는
둘 다 정반대 결론에 다다랐습니다. 에이전트를 사람이 지켜보지 않아도 돌아가게 *해주는* 것은
바로 OS 수준의 진짜 **샌드박스**입니다. 샌드박스는 물어보는 대신 질문 자체를 없애 버리기
때문입니다.

차이를 정확히 정리하면 이렇습니다.

| | **권한(Permissions)** | **샌드박스(Sandboxing)** |
|---|---|---|
| 작동 방식 | 에이전트가 "해도 될까요?" 묻고, 사용자가 예/아니오 클릭 | 커널이 syscall을 거부 — 물어보는 창 자체가 없음 |
| 집행 계층 | 에이전트 자기 코드 (교묘한 프롬프트로 우회 가능) | OS 기본 기능 (Linux는 bubblewrap, macOS는 seatbelt) |
| 하위 프로세스까지 막나? | 아니오 — 에이전트 자신의 tool call만 | 예 — 에이전트가 띄운 스크립트·프로그램·하위 프로세스 전부 |
| 실패 양상 | "클릭 피로" → 사용자가 뭐든 그냥 승인 도장 | 작업 폴더 밖에 쓰기 시도 → syscall에서 `Operation not permitted` |

권한은 에이전트가 던지는 요청이고, 샌드박스는 커널이 세운 벽입니다. 에이전트는 말솜씨로 벽을
넘어갈 수 없습니다. 바로 그래서 그 벽 덕분에 사용자가 감시를 멈출 수 있는 겁니다.

## 두 개의 경계

잘 설계된 코딩 에이전트 샌드박스는 서로 보완하는 두 경계를 세웁니다. 둘 다 OS 수준이라
에이전트의 직접 tool call뿐 아니라 에이전트가 띄운 하위 프로세스까지 덮습니다.

1. **파일시스템 격리** — 현재 작업 폴더 안에서만 읽고 쓸 수 있습니다. 폴더 밖 쓰기는 커널이
   막고(`Operation not permitted`), 클릭할 창도 뜨지 않습니다.
2. **네트워크 격리** — 바깥으로 나가는 트래픽은 샌드박스 *바깥*에서 도는 프록시로 향하는 unix
   domain socket을 통해서만 나갑니다. 프록시가 허용 도메인 정책을 집행하고, 새 도메인은 확인을
   받습니다.

효과는 구체적입니다. Anthropic은 내부 사용에서 샌드박스가 권한 프롬프트를 **84% 줄이면서**
동시에 안전 하한선은 *올렸다*고 보고합니다. 끼어드는 횟수는 줄고 경계는 더 단단해집니다 —
이게 핵심입니다.

## 왜 이게 prompt injection의 정답인가

에이전트 *안*에 사는 prompt injection 방어(탐지, 거부 휴리스틱)는 확률적 필터입니다. 충분히
교묘한 injection은 빠져나갑니다. 샌드박스는 결이 다른 방어입니다. 공격을 *탐지*하려 하지 않고
피해 반경을 *가둡니다*. Anthropic의 표현대로, prompt injection이 성공하더라도 완전히 격리된
상태로 머물러 전체 사용자 보안에는 영향을 주지 못합니다. 샌드박스 안에서 탈취당한 에이전트는
SSH 키를 읽거나 외부로 신호를 보낼 수 없습니다. 작업 폴더 밖 파일을 건드리거나 승인되지 않은
연결을 여는 걸 커널이 허락하지 않기 때문입니다. 모든 injection을 잡아내는 데 거는 대신 OS에
거는 셈인데, 이쪽이 훨씬 좋은 베팅입니다.

## 기본값이 갈린다: 자율성 우선 vs 승인 우선

아키텍처는 거의 똑같습니다. 갈리는 건 *기본값*이고, 여기서 제품 철학이 드러납니다.

- **Claude Code**는 자율성 우선으로 나옵니다. 대부분의 작업은 자유롭게 돌리되, 정말로 파괴적인
  동작에는 여전히 프롬프트를 띄웁니다 (`/`, `$HOME`, 시스템 경로를 겨냥한 `rm`/`rmdir`은
  auto-allow 모드에서도 프롬프트가 뜹니다).
- **Codex**는 승인 우선으로 나옵니다 (신제품에는 가장 안전한 선택). 기본값이 조심스러운 승인
  모드이고, 사용자가 `--full-auto`로 자율성을 켜야 합니다 (= `--approval-mode never` +
  `--sandbox workspace-write`). 단계는 `workspace-write`(기본) → `danger-full-access`(샌드박스
  아예 없음)로 이어집니다.

Codex 문서도 같은 두 축 구분을 명시합니다. **샌드박스는 기술적 경계를 정하고, 승인 정책은 그
경계를 넘기 전에 언제 멈춰 물을지를 정한다.** 슬라이더 하나가 아니라, 독립된 손잡이 둘입니다.

## 정직하게 남겨두는 단서

두 벤더 모두 샌드박스가 완벽한 격리 경계라고 주장하지 않습니다. 가장 중요한 구멍은 이겁니다.
네트워크 프록시는 **TLS 검사를 하지 않습니다**. 어떤 *도메인*에 닿는지는 통제하지만, 허용된
도메인으로 향하는 암호화된 연결 *안에* 무엇이 실려 가는지는 통제하지 못합니다. 그래서
허용됐지만 탈취된 도메인은 여전히 유출 경로가 됩니다. 경계는 "어떤 서버냐"이지 "어떤
데이터냐"가 아닙니다. 도메인 허용 목록을 실질적인 보안 통제 지점으로 여기고, 좁게 유지하세요.

## 그래서 뭘 하나 (실전 정리)

- 코딩 에이전트를 사람 없이 돌리고 싶다면(밤새 티켓 처리, 병렬 worktree), **샌드박스를 먼저,
  자율성 플래그를 나중에** 잡으세요. 반대가 아닙니다. "이제 안 물어봐도 됨"을 안전하게 만드는
  게 바로 샌드박스입니다.
- "프롬프트를 껐다"를 "안전하게 만들었다"와 헷갈리지 마세요. 샌드박스 *없이* 프롬프트만 끄면
  더 나빠지고, 샌드박스와 *함께* 끄면 그게 설계 의도입니다.
- **네트워크 허용 목록**을 진짜 방어선으로 삼으세요. 파일시스템 격리는 쉬운 절반이고, 허용된
  도메인을 통한 유출이 어려운 절반입니다.
- 긴 자율 실행은 프롬프트를 붙잡고 있지 않을 때만 값어치를 합니다. 그러려면 샌드박스가 먼저
  있어야 합니다.
