# 조용한 기본값 변경은 곧 파괴적 동작 변경 — 포스트모템에서 얻은 세 가지 교훈

Anthropic의 [2026년 4월 23일 포스트모템](https://www.anthropic.com/engineering/april-23-postmortem)은 2026년 3~4월에 걸쳐 발생한 세 건의 회귀(regression)를 공개했습니다. 원인은 셋 다 다릅니다. 그런데 개별 버그보다 흥미로운 건 이 셋을 관통하는 공통점입니다. 세 건 모두 **에러를 내지 않고** 출력 품질을 떨어뜨렸고, 그중 둘은 호출하는 쪽 코드는 그대로인데 플랫폼이 알아서 기본값을 바꾼 게 방아쇠였습니다. 여기서 제가 얻은 교훈은 모델 플랫폼 위에 무언가를 만드는 누구에게나 통하는 일반 원칙입니다. **지능에 영향을 주는 기본값은 API 계약의 일부이고, 그걸 소리 없이 바꾸는 건 아무것도 throw하지 않아도 파괴적 변경(breaking change)이라는 것.**

## 첫 번째 원인 — reasoning effort 기본값이 동의 없이 바뀜

**무슨 일이 있었나.** UI가 멈추는 문제를 고치려고 기본 reasoning effort를 `high`에서 `medium`으로 조용히 낮췄습니다. 이 설정을 한 번도 건드린 적 없는 사용자들이 갑자기 복잡한 다단계 작업에서 눈에 띄게 나쁜 결과를 받았습니다. 예전 동작으로 되돌리려면 reasoning 예산을 명시적으로 더 요청해야 했습니다.

**어떻게 고쳤나.** 기본값을 다시 `high`로 되돌렸고, 새 정책을 세웠습니다. 지능에 영향을 주는 기본값은 배포 전에 소크 기간(soak period, 확대된 내부 테스트를 거치는 관찰 기간)과 명시적인 사용자 제어 수단을 반드시 거치도록 한 것입니다.

**교훈.** reasoning effort 설정은 플랫폼 내부 값이 아니라 품질 대 비용을 조절하는 다이얼입니다. 내가 한 번도 안 건드려도, 매 요청마다 기본값이 일을 하고 있으니 사실상 내가 지정한 파라미터처럼 동작합니다. 이걸 조용히 바꾸면 API 동작이 바뀐 겁니다. 그래서 모델 플랫폼 위에 무언가를 만든다면 **usage 응답에 찍히는 thinking token 수를 봐야 합니다.** 내 코드는 그대로인데 이 값이 뚝 떨어졌다면 플랫폼 쪽에서 effort를 조정한 흔적입니다. 출력이 실패하는 게 아니라 은근히 나빠질 뿐이라, 이게 관측 가능한 유일한 신호인 경우가 많습니다.

## 두 번째 원인 — 대화 중간에 reasoning 상태가 소리 없이 지워짐

유휴 reasoning 블록을 한 번만 정리하려던 최적화가 실수로 **매 턴마다** 발동했습니다. 긴 세션이 이어지는 동안 모델 자신의 reasoning 체인을 조금씩 지워버린 겁니다. 이 실패 양상이 이 글의 주제를 가장 순수하게 보여줍니다. 에러도 예외도 없이, 세션이 길어질수록 조용히 점점 더 앞뒤가 안 맞아집니다. 딛고 서 있던 reasoning이 밑에서 계속 삭제되고 있었으니까요.

여기서 감지 신호는 구조적입니다. 긴 세션에서 앞뒤 안 맞는 전개가 나타나고, reasoning을 해야 할 턴인데 thinking token이 0으로 찍히는 겁니다. 예방책은 **reasoning 블록을 대화 히스토리에 직접 보존하고 매 턴 존재 여부를 확인하는 것**입니다. 플랫폼이 알아서 유지해 주겠거니 하고 넘겨짚지 마세요.

## 세 번째 원인 — 장황함 제한이 모델 자신의 계획 수립을 잘라먹음

**무슨 일이 있었나.** 응답을 덜 장황하게 만들려고 하드한 단어 수 제한을 담은 시스템 프롬프트를 추가했습니다. 대략 tool call 사이는 25단어, 최종 응답은 100단어였습니다.

**결과.** 내부 eval에서 코딩 품질이 측정 가능하게 떨어졌습니다. 단어 수 제한 탓에 reasoning 단계가 도중에 잘려 나갔고, 버그와 엣지 케이스를 잡아냈을 분석까지 함께 끊긴 겁니다.

**어떻게 고쳤나.** 단어 수 제한 시스템 프롬프트를 제거했습니다. 그리고 장황함 제어를 다른 층위에서 하도록 재설계했습니다. 중간 단계에 인라인 token 예산을 거는 대신 output style 제약으로 옮긴 겁니다.

**교훈.** 이건 직접 만들 때 가장 헷갈리기 쉬운 대목입니다. **중간** reasoning — tool call 사이의 코멘터리, 단계와 단계 사이의 사고 과정 — 에 거는 token 예산은 출력 장황함 제어와 **다른 것**입니다. 둘 다 "더 짧게 만들기"처럼 보이는데도 그렇습니다. "tool call 사이 단어 수"를 제한하면 산문만 줄어드는 게 아니라 모델의 계획 수립 자체가 잘립니다. 요약이 아니라 사고를 자르는 거라, 품질 타격은 직관보다 큽니다. 출력을 더 간결하게 하고 싶으면 최종 답변의 형식을 제약하세요. 거기까지 가는 단계에 단어 수 상한을 걸지 마세요.

## 관통하는 패턴

| 실패 유형 | 감지 신호 | 예방책 |
|---|---|---|
| reasoning effort가 바뀜 | usage의 thinking token 수가 예고 없이 떨어짐 | 모델 버전 간 usage 변화량에 알림 걸기 |
| reasoning 상태가 사라짐 | 긴 세션에서 앞뒤 안 맞는 전개, reasoning 턴인데 thinking token이 0 | reasoning 블록을 히스토리에 보존하고 매 턴 존재 확인 |
| 중간 단계에 걸린 장황함 제한 | 코드 품질 지표가 나빠지고 응답이 생각 도중에 끊긴 느낌 | 장황함 제어와 reasoning 예산을 분리하고 양쪽 다 eval로 테스트 |

공통점은 이겁니다. **셋 중 어느 것도 에러를 던지지 않았습니다.** 전부 조용히 품질만 떨어뜨렸다는 건, 감지 전략이 "예외를 잡는다"가 될 수 없다는 뜻입니다. "텔레메트리를 본다"여야 합니다. 모델 버전 간 usage 변화량이 곧 연기 감지기입니다. 코드는 안 바꿨는데 출력 품질이 흘러내리고 있다면, 내 프롬프트를 뜯어보기 전에 먼저 thinking token 수와 턴별 reasoning 존재 여부부터 보세요.

이 대응에서 한 플랫폼을 넘어 일반화할 만한 게 두 가지 있습니다.

- **소크 기간은 기능 변경만이 아니라 지능에 영향을 주는 변경까지 포함해야 합니다.** 깔끔하게 배포된 변경(크래시도 에러도 없는)이라도 품질 다이얼을 조용히 움직였다면 회귀입니다. 기본 effort 변경은 모든 기능 게이트를 통과하고도 출력을 떨어뜨렸습니다.
- **시스템 프롬프트를 바꿀 땐 같은 eval 스위트를 변경 전후 양쪽에 돌리세요.** 장황함 제한은 프롬프트 수정으로만 보면 무해해 보였는데 실제 품질 비용을 냈습니다. 프롬프트 변경도 코드 변경과 같은 eval 규율이 필요합니다.
